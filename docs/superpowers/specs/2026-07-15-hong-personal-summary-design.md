# 洪振博个人总结报告设计

## 目标

基于 `CLAUDE.md`、项目源码、当前 Git 历史、服务器/数据库配置和已有《团队总结.docx》，生成一份独立的《洪振博个人总结报告.docx》。报告面向课程设计提交和答辩准备，重点说明洪振博在基础架构、竞赛管理、Bug 修复、服务器部署、数据库管理、前端优化和团队协作中的实际贡献。

## 写作口径

1. 以代码、提交记录和项目文档为主要证据，不虚构不存在的测试数据。
2. 对个人独立完成、协作完成和团队其他成员完成的工作分别表述。
3. 将“已完成”“开发过程中已验证”“当前仍需完善”分开，尤其说明当前工作区存在的未提交上传/图片相关修改。
4. 不在报告中暴露数据库密码或其他敏感凭据；服务器地址只作为部署环境背景记录。
5. 报告时间口径为 2026 年 7 月 15 日，项目周期为 2026 年 7 月 4 日至 7 月 19 日。

## 正文结构

1. 封面与基本信息。
2. 摘要：概括个人职责、核心成果和工程化价值。
3. 项目背景与个人角色：说明系统目标、技术栈、洪振博的副队长/模块负责人定位。
4. 基础架构与数据库设计：说明 MVC、Maven、Java EE/Tomcat 9 适配、数据库表结构、SQL 脚本和统一规范。
5. 竞赛管理模块：说明 DAO、Service、Servlet、JSP、分类管理、输入校验、RBAC 和跨模块状态展示。
6. Bug 修复与稳定性治理：按问题、原因、修复、效果记录认证、权限、JSP 500、作品截止、图片访问、缩略图、获奖列表和入队申请等问题。
7. 服务器部署与数据库管理：说明 WAR 构建、8080 端口、Tomcat 配置、远程 MySQL 切换、schema/data/migration 脚本、外置上传目录和部署注意事项。
8. 前端与交互优化：说明角色首页、统一导航、页面族样式、React + Vite + GSAP Landing 页和响应式/无障碍考虑。
9. Git 协作与个人工作量证据：说明 70 次署名提交（其中 62 次非合并提交）、代表性提交和开发时间线；不把提交次数等同于代码质量。
10. 个人能力提升与团队协作：总结需求拆解、跨模块联调、冲突处理、部署维护和文档意识。
11. 不足、风险与后续计划：包括凭据外置、完整自动化验收、旧 BLOB 数据、工作区未提交改动和环境复现等问题。
12. 结语与证据索引：列出关键文件、提交 ID 和核验说明。

## 版式

使用中文课程报告风格：A4 纵向、宋体正文、黑体分级标题、统一页眉页脚、封面、目录式章节列表、表格化贡献清单和问题修复清单。生成独立文件，不覆盖用户提供的 `C:\Users\Lenovo\Desktop\团队总结.docx`。

## 验收标准

- DOCX 可以被 `python-docx` 重新打开，正文和表格内容完整。
- 文档中包含 Bug 修复、服务器部署和服务器数据库管理三个独立部分。
- 文档不包含数据库密码。
- 报告中的 Git 数量、项目文件统计和关键提交与当前仓库核对结果一致。
- 原有源代码和用户当前工作区修改不被覆盖。
